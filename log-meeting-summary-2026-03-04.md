# 2026-03-04 Log 專案會議摘要

## 1. 專案目標與現況
*   **核心目標**：在 ELK 與 APM 平臺上實作完整的 Log 記錄與效能追蹤。
*   **當前進度**：
    *   已完成第一階段測試環境（Windows + ELK/APM）架設，成功接收 .NET 範例程式 Log。
    *   正在進行 Windows、Linux 及 IIS 日誌的盤點與收容規劃。

## 2. 環境與基礎建設
*   **容器環境轉型**：
    *   開發/測試環境目前使用 Rocky Linux 9.7 作為底層。
    *   未來正式環境預計改用 **Red Hat (RHEL)**，以便取得原廠授權與技術支援。
    *   排除 164 主機防火牆連線問題，確保 Log 能順利往測試環境傳送。
*   **資源優化議題**：
    *   檢討現有主機數量是否過多。由於現代 64 位元架構與 .NET 4.8 的效能提升，考慮減少主機數量並提升單機規格，以降低維運成本與作業系統損耗。

## 3. 技術規格與標準
*   **OpenTelemetry (OTEL)**：確認未來將朝 OTEL 標準前進，確保 Log 與 APM 的通用性，避免被特定廠商鎖定。
*   **APM 版本控管**：規範 APM Agent 版本（如 1.22），避免版本不一導致 Trace ID 追蹤斷鏈。
*   **.NET 升級**：確認 NIS 等系統需升級至 **.NET Framework 4.8** 以完整支援 Log 收容功能。

## 4. 跨系統追蹤 (Distributed Tracing)
*   **Trace ID 實作**：確認 APM 可自動串接跨主機（如 IT105 到 IT223）的 Request，解決以往過了 Load Balancer 就難以追蹤的問題。
*   **業務代號查詢**：未來可在 Dashboard 實作模糊查詢（如保單號碼、案號），透過業務資料反查整條交易鏈。

## 5. 待辦事項 (Action Items)
*   **系統科**：提供 Windows/Linux/IIS 日誌盤點表。
*   **資安科**：與數聯研討 IIS 加密 Log 拋轉至 SOC 的格式問題，並整理安裝手冊。
*   **架構組**：
    *   在測試環境（179 主機等）進行 .NET 4.8 分批升級測試。
    *   將 SKB 工具與 APM 功能推送到開發環境進行試作。
    *   評估使用 OTL Collector 轉發日誌的可行性。
